## 2. 产品定位

Virtual Office 不是一个“会动的虚拟办公室”，而是一个以空间化界面承载的多角色 Agent 公司操作系统。

系统通过“虚拟办公室 + Agent 编排 + 设备执行层 + 权限治理”的方式，把产品、研发、运营、HR、采购等职能角色映射为可协作的数字员工。每个角色都是一个具备目标、权限、记忆和工具能力的 Agent，并在需要时调度真实执行资源，例如 Kubernetes OpenClaw runner、浏览器、终端、IDE 或内部系统账户。

## 3. 背景与机会

多数企业已经开始使用 AI，但仍然存在以下问题：

- AI 只停留在个人工具层，无法形成组织级协同
- 多个职能之间仍然靠聊天、会议和人工同步推动工作
- 关键任务执行与真实设备操作脱节，无法闭环
- 高级员工成为审批与知识瓶颈，系统不可扩展
- 管理层看不到 Agent 的真实负载、协作状态和风险位置

Virtual Office 的目标是把“组织如何工作”变成一个可视化、可调度、可审计、可扩展的系统。

## 4. 一句话定义

构建一个以空间化 UI 展示组织运行状态、以多角色 Agent 承担执行、以 Kubernetes OpenClaw runner 等真实设备完成落地操作的人机协同公司操作系统。

## 5. 产品目标

### 5.1 业务目标

- 缩短跨职能任务从发起到完成的周期
- 提升组织中可自动化工作的覆盖率
- 降低重复性沟通与人工协调成本
- 提高对真实执行过程的透明度和可控性

### 5.2 组织目标

- 让每个核心职能都有可用的 Agent 代理角色
- 让人类从“执行者”转为“目标设定者、批准者和例外处理者”
- 让跨职能工作从串行沟通转为共享上下文下的并行协作
- 让真实设备执行成为可观测、可治理的基础能力

### 5.3 北极星指标

- 单个跨职能任务的平均流转时长下降 50%
- 低风险任务的自动执行占比达到 60% 以上
- 关键任务的状态透明度达到 100%
- 设备执行成功率达到 95% 以上

## 6. 非目标

- 不做单纯展示型办公室动画
- 不做完全无人值守的公司
- 不让所有角色永久绑定专属设备
- 不在没有审批和审计的情况下执行高风险业务动作
- 不在第一阶段接入财务支付、合同签署等最高风险流程

## 7. 核心设计原则

- 角色与设备解耦：角色是逻辑身份，设备是可调度资源
- 空间是界面，不是核心：真正核心是任务、状态、执行和治理
- 所有重要动作必须可追溯
- 高风险动作必须显式审批
- 人类负责目标、边界、风险和最终判断
- Agent 负责执行、协作、分析、整理和初步决策

## 9. 核心概念

### 9.1 角色 Agent

具备明确职能身份、目标范围、工具权限、记忆上下文和协作方式的逻辑实体。

示例：

- 产品 Agent
- 研发 Agent
- 运营 Agent
- HR Agent
- 采购 Agent
- CEO Office Agent

### 9.2 空间节点

虚拟办公室中的可视化区域，用于承载某类角色、任务或设备状态，例如研发区、会议区、指挥中心、设备机房。

### 9.3 任务

所有工作都必须被结构化为任务，任务包含目标、上下文、输入、输出、风险等级、审批要求和执行记录。

### 9.4 设备资源

可被 Agent 动态申请和释放的执行资源，例如：

- Kubernetes OpenClaw runner
- 浏览器会话
- IDE 会话
- 远程桌面
- 受控业务系统账号

### 9.5 执行会话

一次 Agent 使用某台设备完成某项任务的生命周期记录，包含开始时间、操作日志、截图、产出物和结束状态。

### 9.6 交接

一个 Agent 将任务、上下文或产物移交给另一个 Agent 的过程，例如产品交给研发、研发交给运营、HR 交给行政。

### 9.7 审批

高风险动作进入人工批准环节，例如：

- 发布到生产环境
- 修改权限
- 发送正式对外通知
- 创建采购单
- 推进招聘 Offer

## 10. 角色范围

## 10.1 第一阶段角色

- 产品 Agent：需求澄清、PRD 草案、优先级建议、验收标准整理
- 研发 Agent：编码、测试、调试、PR、部署准备
- 运营 Agent：发布同步、活动配置、数据整理、复盘草稿
- CEO Office Agent：跨部门协调、日报周报、阻塞升级
- HR Agent：候选人流转、面试协调、入离职流程跟进
- 采购 Agent：需求收集、供应商对比、采购建议和审批发起

## 10.2 第二阶段角色

- 财务 Agent
- 客服 Agent
- 法务 Agent
- 安全 Agent
- 数据分析 Agent

## 11. 用户场景

### 场景一：产品到研发到运营的发布闭环

1. 产品 Agent 创建需求并整理背景
2. 研发 Agent 接收任务并申请一个 Kubernetes OpenClaw runner 执行开发
3. 研发 Agent 完成代码、测试和提交流程
4. 运营 Agent 读取发布结果并生成发布说明
5. 管理者在虚拟办公室中观察全过程和关键节点状态

### 场景二：HR 招聘流转

1. HR Agent 创建候选人任务
2. Agent 自动协调面试、汇总反馈、生成进展摘要
3. 高风险节点如 Offer 发出需人工审批
4. 所有状态在虚拟办公室中以角色和任务形式展示

### 场景三：采购流转

1. 采购需求进入系统
2. 采购 Agent 收集规格、比价、整理供应商资料
3. 预算和下单动作进入审批
4. 批准后由受控设备执行后续流程

## 12. 功能范围

## 12.1 P0 功能

### A. 虚拟办公室主视图

- 展示办公室地图和职能区域
- 展示角色 Agent 的位置、状态和任务气泡
- 展示关键指标摘要，例如待办、执行中、阻塞、异常

### B. 角色详情面板

- 显示角色身份、当前任务、能力标签、运行状态
- 显示最近行为、任务历史、协作对象
- 显示当前绑定或待申请的设备资源

### C. 任务系统

- 创建任务
- 指派角色
- 任务流转
- 任务状态变更
- 任务交接记录

### D. 设备资源调度

- 显示在线 Kubernetes OpenClaw runner 列表
- 显示设备状态、资源占用、健康状况
- 支持 Agent 为任务申请执行会话
- 支持任务结束后释放设备

### E. 审批与人工接管

- 高风险任务进入审批队列
- 支持人工接管执行会话
- 支持暂停、终止、重试和改派

### F. 实时事件流

- 显示 Agent 执行事件
- 显示设备异常
- 显示关键任务升级
- 显示协作交接

## 12.2 P1 功能

- 多层地图与楼层视图
- Agent 群组协同与会议模式
- 任务自动拆解和跨角色自动编排
- 成本与 token 消耗视图
- Agent 长期记忆和个性化工作偏好
- 按部门或项目切换视图

## 12.3 P2 功能

- 多办公室 / 多组织支持

## 13. 关键交互设计

### 13.1 首页

以等距或空间化地图呈现办公室，不追求游戏玩法，重点呈现：

- 角色状态
- 部门区域
- 任务密度
- 告警热点
- 设备负载

### 13.2 指挥中心

提供列表和控制台视图，面向管理者和操作员：

- 全局任务视图
- 全局设备视图
- 审批队列
- 异常事件流
- 人工接管入口

### 13.3 角色抽屉

- 当前任务
- 最近动作
- 关联文档
- 绑定设备
- 风险提示

### 13.4 设备抽屉

- 设备名称
- 所在机房 / 所属资源池
- CPU / 内存 / 在线状态
- 当前会话
- 最近执行日志

## 14. 关键流程

### 14.1 任务创建流程

1. 用户或 Agent 创建任务
2. 系统根据任务类型推荐目标角色
3. 任务进入角色待办或自动编排队列
4. 若需要真实执行，角色申请设备会话

### 14.2 Agent 执行流程

1. Agent 读取任务上下文
2. Agent 申请工具和设备
3. Agent 执行任务并持续上报状态
4. 系统记录日志、截图、产出物
5. 成功、失败或升级后结束会话

### 14.3 交接流程

1. 当前角色完成本阶段工作
2. Agent 生成结构化交接摘要
3. 下一个角色接收任务和上下文
4. 交接事件进入事件流和审计记录

### 14.4 审批流程

1. 任务命中审批规则
2. 系统暂停关键动作
3. 人类查看风险摘要与建议动作
4. 选择批准、驳回、修改后重试或人工接管

## 16. 风险与约束

- 如果只做 UI，不做任务和执行闭环，产品会变成演示系统
- 如果角色永久绑定设备，资源成本和扩展复杂度会迅速上升
- 如果缺少权限、审批和审计，高风险职能无法真正上线
- 如果没有结构化任务系统，Agent 之间会产生大量无效交接
- 如果没有人工接管机制，系统出错时不可控

## 17. MVP 建议

第一阶段建议只覆盖“产品 + 研发 + 运营 + CEO Office”四类角色，并只打通一条链路：

1. 产品创建需求
2. 研发 Agent 绑定 Kubernetes OpenClaw runner 执行开发和测试
3. 系统展示真实进度和状态
4. 运营 Agent 生成发布说明
5. 管理者在 UI 中完成批准与观察

MVP 必须做到：

- 不是静态页面
- 不是纯文本任务系统
- 必须有真实设备执行
- 必须有实时状态流
- 必须有人工审批和接管
